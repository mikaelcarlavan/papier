Forms

An AcroForm turns a static page into an interactive document: fields the reader can type into, tick, or choose from. Papier builds forms from scratch through the Papier\AcroForm namespace, and fills forms in existing PDFs through FormFiller.

How AcroForms work

A PDF form is two structures that must agree with each other. The field tree lives in the document catalog under /AcroForm and describes the form logically: names, types, values, and validation flags. The widget annotations live on individual pages and describe where each field is drawn and what it looks like.

Papier keeps these in sync for you, but it does mean every field needs to be registered twice — once with the form and once with the page it appears on:

use Papier\AcroForm\{AcroForm, TextField};
use Papier\PdfDocument;

$doc  = PdfDocument::create();
$font = $doc->addFont('Helvetica');   // returns a resource name such as 'F1'
$page = $doc->addPage();

$form = new AcroForm();

$field = new TextField('reviewer', 'reviewer');
$field->setRect(200, 700, 400, 718)
      ->setDefaultAppearance($font, 12);

$form->addField($field);      // logical: joins the /AcroForm /Fields array
$page->addFormField($field);  // visual:  emits the widget annotation on this page

$doc->setAcroForm($form);
$doc->save('form.pdf');

Forgetting $page->addFormField() is the single most common mistake. The field will exist in the document and be readable by a parser, but nothing will be visible or clickable on the page.

Field names

Every field constructor takes two names. The first is the fully-qualified name — the dotted path used to address the field programmatically and in form submissions. The second is the partial name, written to /T, which is the field's own name within its parent.

$field = new TextField('person.name', 'name');

For flat forms the two are usually the same, and passing just one argument makes the partial name default to the full name:

$field = new TextField('email', 'email');
// or simply, since partialName defaults to the full name:
$field = new TextField('email');

Dotted names matter when you have repeating groups — invoice.line1.qty, invoice.line2.qty — because form-filling tools and JavaScript address fields by their fully-qualified name.

Positioning and appearance

All fields inherit from FormField, which supplies the geometry and styling that every widget needs.

setRect() takes the lower-left and upper-right corners of the widget in points, in the page's default user space (origin at the bottom-left of the page):

$field->setRect($x1, $y1, $x2, $y2);

setDefaultAppearance() writes the /DA string that tells the viewer which font, size, and colour to use when it renders the field's value. Note that the first argument is the font resource name string returned by addFont(), not a font object:

$helv = $doc->addFont('Helvetica');       // 'F1'

$field->setDefaultAppearance($helv, 11);              // black text
$field->setDefaultAppearance($helv, 11, 0.1, 0.1, 0.5); // dark blue, RGB in [0,1]

If you need a /DA string that Papier does not generate, write it directly with setDefaultAppearanceRaw('/F1 10 Tf 0 g').

Widget chrome — the box drawn around the field — is controlled separately:

use Papier\Elements\Color;

$field->setBackgroundColor(Color::rgb(0.96, 0.96, 1.0))
      ->setBorderColor(Color::gray(0.6))
      ->setHighlightMode('P');   // N=none, I=invert, O=outline, P=push, T=toggle

Both colour setters also accept loose components: setBorderColor(0.6, 0.6, 0.6).

Values and flags

Field values are PDF objects rather than plain PHP scalars, because the correct object type depends on the field type — a text value is a string, a checkbox value is a name.

use Papier\Objects\PdfString;

$field->setValue(PdfString::text('Jane Doe'));
$field->setDefaultValue(PdfString::text('')); // /DV — what "Reset form" restores

Use PdfString::text() rather than new PdfString() for anything a human will read: it applies the correct PDF text-string encoding, so accented and non-Latin characters survive.

Behavioural flags live in the /Ff bit field. The two most common have convenience wrappers:

$field->setReadOnly()    // display the value but forbid editing
      ->setRequired();   // block submission while empty

Anything else is set with setFlags(), using the constants published by each field class:

use Papier\AcroForm\TextField;

$field->setFlags(TextField::FLAG_MULTILINE | TextField::FLAG_DO_NOT_SPELL_CHECK);

setFlags() replaces the whole flag word rather than merging into it. Combine every flag you want in a single call, and set it before calling helpers like setMultiline() that also touch /Ff.

Two more inherited helpers are worth knowing: setToolTip() writes /TU, the alternate description read aloud by screen readers (see Tagged PDF & PDF/A), and setMappingName() writes /TM, the name used when exporting form data.

Text fields

TextField is the workhorse: a single- or multi-line box for free-form input.

use Papier\AcroForm\TextField;

$comments = new TextField('Comments', 'Comments');
$comments->setMultiline(true)
         ->setDefaultAppearance($font, 10)
         ->setRect(72, 430, 523, 492);

Beyond setMultiline(), the class offers setMaxLength(), setPassword() (masks input, and prevents the value being saved), and setRichText(). The flag constants let you reach the rest:

ConstantEffect
FLAG_MULTILINEText wraps across several lines
FLAG_PASSWORDInput is masked and never stored
FLAG_FILE_SELECTThe value is a file path to upload
FLAG_DO_NOT_SPELL_CHECKSuppress the viewer's spell checker
FLAG_DO_NOT_SCROLLReject input once the box is full
FLAG_COMBEvenly spaced character cells (needs setMaxLength())
FLAG_RICH_TEXTThe value carries formatting markup

A comb field — the boxed one-character-per-cell style used on paper forms — needs both the flag and a maximum length:

$postcode = new TextField('postcode');
$postcode->setMaxLength(8)
         ->setFlags(TextField::FLAG_COMB)
         ->setDefaultAppearance($font, 12)
         ->setRect(72, 600, 220, 620);

Check boxes

CheckBoxField is a two-state button. Its "on" state is a named export value — Yes by default — and its "off" state is always the reserved name Off.

use Papier\AcroForm\CheckBoxField;

$news = new CheckBoxField('Newsletter', 'Newsletter');
$news->setRect(140, 568, 155, 583);
$news->check();     // value becomes /Yes
$news->uncheck();   // value becomes /Off

The export value is what appears in submitted form data, so give it a meaningful name when the form is going to be processed:

$news = new CheckBoxField('Newsletter', 'Newsletter', 'subscribed');
$news->check();                       // value becomes /subscribed
echo $news->getOnValue();             // 'subscribed'

The class name is CheckBoxField, with a capital B. Check boxes are square by convention and radio buttons round, but that is a viewer rendering convention, not something the field type enforces.

Radio buttons

A radio group is one RadioButtonField whose kids are the individual buttons. Because they share a parent, selecting one deselects the others — that mutual exclusion is a property of the field tree, not of the widgets.

use Papier\AcroForm\RadioButtonField;

$size = new RadioButtonField('Size', 'Size');
$size->setSelected('medium');   // the export value that starts selected
$form->addField($size);

Attach the individual buttons as kids with addKid(), giving each its own rectangle and export value.

Choice fields

ComboBoxField (a drop-down) and ListBoxField (a scrolling list) are both choice fields. Each option has a machine value and a human label, and both classes require a call to buildOpt() to serialise the collected options into the /Opt array.

use Papier\AcroForm\ComboBoxField;

$country = new ComboBoxField('Country', 'Country');
$country->addOption('US', 'United States')
        ->addOption('GB', 'United Kingdom')
        ->addOption('FR', 'France')
        ->buildOpt();                        // required — writes /Opt

$country->setDefaultAppearance($font, 10);
$country->setRect(140, 600, 280, 622);

A list box works the same way and adds multi-selection:

use Papier\AcroForm\ListBoxField;

$lang = new ListBoxField('Language', 'Language');
$lang->addOption('en', 'English')
     ->addOption('fr', 'French')
     ->addOption('de', 'German')
     ->setMultiSelect(true)
     ->buildOpt();

$lang->setRect(375, 590, 523, 635);

Call buildOpt() after the last addOption(). Options added afterwards will not appear in the output.

Combo boxes accept extra flags: ComboBoxField::FLAG_EDIT allows the user to type a value that is not in the list, and FLAG_SORT asks the viewer to sort the options alphabetically.

Push buttons

A PushButtonField holds no value — it exists purely to trigger an action. Papier does not paint the button face for you, so draw it with ordinary page elements and overlay the field:

use Papier\AcroForm\PushButtonField;
use Papier\Elements\{Color, Rectangle, Text};

$submit = new PushButtonField('Submit', 'Submit');
$submit->setRect(310, 380, 430, 405);
$form->addField($submit);
$page->addFormField($submit);

$page->add(
    Rectangle::create(310, 380, 120, 25)->fill(Color::rgb(0.2, 0.5, 0.9)),
    Text::write('Submit Form')->at(335, 390)->font($bold, 12)->color(Color::white()),
);

To make the button do something, attach an action to its widget. SubmitFormAction posts the form data to a URL, ResetFormAction restores every field to its /DV, and JavaScriptAction runs a script. See Bookmarks & navigation for the full action catalogue.

Signature fields

A SignatureField reserves a place for a digital signature. Created empty it is an unsigned placeholder that a signing tool can later fill:

use Papier\AcroForm\SignatureField;

$sig = new SignatureField('Signature', 'Signature');
$sig->setRect(360, 690, 530, 750);
$form->addField($sig);
$page->addFormField($sig);

$form->setSigFlags(3);   // 1 = SignaturesExist, 2 = AppendOnly

setSigFlags(3) tells viewers that the document contains signatures and must only be modified by appending, which is what keeps existing signatures valid. To actually apply a cryptographic signature, use PdfSigner — see Encryption & signatures.

Form-level settings

The AcroForm object carries defaults and document-wide switches that apply across every field.

use Papier\AcroForm\AcroForm;
use Papier\Objects\PdfDictionary;

$form = new AcroForm();
$form->setNeedAppearances(true);        // ask the viewer to render field values
$form->setDefaultAppearance($font, 10); // fallback /DA for fields without one
$form->setQuadding(0);                  // 0 = left, 1 = centre, 2 = right

$resources = new PdfDictionary();
$resources->set('Font', new PdfDictionary());
$form->setDefaultResources($resources); // /DR — resources the /DA strings refer to

setNeedAppearances(true) is the pragmatic choice when you are authoring a form to be filled in later: it instructs the viewer to generate appearance streams from the /DA string and the current value, so a field always displays correctly even if no appearance was baked in. The trade-off is that some minimal or embedded viewers ignore the flag, and a document whose appearance depends on the viewer is not suitable for archiving — see Tagged PDF & PDF/A.

setCalculateOrder() takes an array of field references and writes /CO, which fixes the order in which calculation scripts run when fields depend on each other.

Filling an existing form

Authoring a form and filling one are different problems. FormFiller opens an existing PDF, sets values, and writes the result back as an incremental update — the original bytes are preserved and a new revision is appended. That keeps any prior digital signatures intact and makes the change auditable.

use Papier\AcroForm\FormFiller;

$filler = new FormFiller(file_get_contents('form.pdf'));

print_r($filler->getFieldNames());   // discover what the form contains

$filler->setText('name', 'Alice Example')
       ->setText('email', 'alice@example.com')
       ->setCheckbox('subscribe', true);

$filler->saveAs('filled.pdf');

Use save() instead when you want the PDF bytes back rather than a file on disk — handy for streaming a filled form straight to an HTTP response:

$pdf = $filler->save();   // returns a string
file_put_contents('filled.pdf', $pdf);

The three setters map onto the field types:

MethodUse forNotes
setText($name, $value)Text fieldsRegenerates the appearance stream from the field's /DA
setCheckbox($name, $on)Check boxes, radio buttonstrue selects the on-value; pass a string to pick a specific export value
setValue($name, $value)Anything elseLower-level escape hatch for choice fields and unusual types

The important detail is that setText() regenerates the appearance stream rather than relying on /NeedAppearances. The value is laid out into a form XObject using the field's own default appearance, so the filled text is visible in every viewer — including ones that ignore /NeedAppearances, and in print output and thumbnails.

Field names must match exactly, including case and any dotted prefix. Always call getFieldNames() first when working with a form you did not author.

Reading values back

To inspect a form without modifying it, use the parser directly:

use Papier\Parser\PdfParser;

$parser = new PdfParser(file_get_contents('filled.pdf'));
$parser->parse();

foreach ($parser->getFormFields() as $field) {
    echo "{$field['name']} ({$field['type']}) = {$field['value']}\n";
}

Each entry reports the field's fully-qualified name, its type (Tx, Btn, Ch, or Sig), and its current value. This is the natural way to verify a filled form or to extract submitted data server-side. More parsing recipes are in Reading PDFs.
