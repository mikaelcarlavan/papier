Examples

The examples/ directory contains runnable, self-contained scripts — one per feature area. They are the fastest way to see a working call sequence, and they double as the library's integration test suite.

Running an example

Every script is a plain PHP file with no arguments and no configuration. Install dependencies once, then run whichever script interests you:

composer install
php examples/01_hello_world.php

Each script prints what it created and writes its output to examples/output/, creating that directory if it does not exist. Nothing is written anywhere else, so you can delete examples/output/ at any time to start clean.

Some scripts also print diagnostics rather than only producing a file. Example 17 reports the size difference between a classic and a compressed build; example 18 proves that the original bytes survive an incremental update unchanged; example 29 prints the text it extracted so you can compare it to what was written.

Examples 15, 20 and 21 embed font files. Lato-Regular.ttf ships in examples/; the CJK examples fall back gracefully if a system CJK font is not found, so they still run on a bare machine.

Verifying the output

Producing a file that parses is not the same as producing a file that displays. A PDF can be structurally valid and still render as a blank page if a resource name is wrong or a content stream is malformed.

The verification tool closes that gap:

php tools/verify.php

It runs every example, renders each resulting PDF with Ghostscript, and runs qpdf --check over it. Ghostscript catches content-level problems — a page that raster­ises to nothing, an undefined operator, a missing font resource. qpdf --check catches structural problems — bad offsets, broken object streams, an inconsistent cross-reference table, damaged filters. Between them they confirm that the files a change produces are not merely well-formed but actually usable.

Run it after any change to the writer. It needs gs and qpdf on your PATH; both are available from Homebrew and every mainstream Linux package manager.

For archival output there is also a validator you can call from your own code:

use Papier\Validation\PdfAValidator;

$issues = PdfAValidator::validate($pdfBytes);   // [] means structurally conformant

The signature examples generate a throw-away self-signed certificate at run time so the script works with no setup. That certificate is not trusted by any viewer — the signature will show as "signer unknown". Supply a real certificate and key for production use.

Using an example as a starting point

The scripts are written to be copied. Each begins with a comment block stating what it demonstrates and which section of ISO 32000-1 applies, uses grouped use statements so the imports are easy to lift, and keeps setup and output separated by section comments.

The most common adaptation is to swap the output. Every example calls save() or toString() near the end:

$doc->save("$outDir/01_hello_world.pdf");   // write to disk

$bytes = $doc->toString();                  // or keep it in memory

Use toString() when you are streaming a response, storing the document in a database, or passing it straight to PdfParser — as several of the examples do to verify their own output without touching the filesystem.
