Reading PDFs

Papier is not only a writer. Papier\Parser\PdfParser opens an existing PDF — including encrypted, compressed and damaged ones — and exposes its pages, text, fonts, images, metadata and interactive content through a small set of plain-array getters.

Parsing a file

The parser takes the complete file contents as a string, not a path. That keeps it usable with data from a database column, an HTTP upload, or an S3 object, without a temporary file.

use Papier\Parser\PdfParser;

$parser = new PdfParser(file_get_contents('document.pdf'));
$parser->parse();

echo $parser->getPageCount() . " pages\n";
echo $parser->getVersion() . "\n";   // e.g. '1.7'

parse() is the mandatory first step. It locates startxref, reads the cross-reference section and trailer, sets up decryption if the document is encrypted, and resolves the catalog and Info dictionary. Every getter below assumes it has already run. Calling it once is enough — objects are cached as they are resolved, so repeated getters do not re-read the file.

Encrypted documents

If the document is password-protected, supply the password before calling parse(). Papier decrypts RC4 (40- and 128-bit), AES-128 and AES-256 transparently; from that point onwards every string and stream you touch is already in the clear.

$parser = new PdfParser(file_get_contents('protected.pdf'));
$parser->setPassword('secret');
$parser->parse();

echo $parser->extractText();

Either the user password or the owner password works — the owner password additionally bypasses the permission flags. An empty password is the common case for documents that are encrypted for permissions only (no open password), and it is what you get by default, so those files need no special handling at all.

To find out whether a document is encrypted before deciding what to do:

if ($parser->isEncrypted()) {
    $parser->setPassword($passwordFromUser);
}
$parser->parse();

setPassword() returns $this, so it chains: (new PdfParser($bytes))->setPassword('secret'). See Security for the writing side.

Text extraction

Extracting text from a PDF is harder than it looks. A PDF does not store paragraphs or even words — it stores instructions like "set font F3 at 11 pt, move to (72, 700), show these bytes". Recovering readable text means interpreting the content stream, tracking the text state, mapping the shown byte codes back to Unicode, and inferring word and line breaks from geometry.

$all   = $parser->extractText();                  // every page, joined
$page1 = $parser->extractTextFromPageNumber(1);   // one page, 1-based

extractText() accepts a separator used between pages, which defaults to a newline:

$text = $parser->extractText("\n\n--- page break ---\n\n");

If you already hold a page dictionary — for instance while walking getPages() yourself — you can extract from it directly with extractTextFromPage($pageDict).

ToUnicode awareness

The byte 0x41 in a content stream does not necessarily mean the letter "A". What it means depends on the font's encoding, and for subset or composite fonts the codes are frequently arbitrary — a subsetted font may map glyph 3 to "é" and glyph 4 to "ü" with no relationship to any standard encoding.

Papier resolves this the way the specification intends: it reads each font's /ToUnicode CMap, which is the reverse map from character codes to Unicode code points that a well-formed PDF is required to supply for exactly this purpose. Where no /ToUnicode exists it falls back to the font's /Encoding (WinAnsi, MacRoman, or the base encoding with any /Differences applied).

This is what makes accented Latin text, subsetted fonts, and Type 0 / CIDFontType2 composite fonts — the ones used for CJK, Cyrillic and Greek — extract as correct UTF-8 rather than as mojibake. Because Papier's own writer generates /ToUnicode automatically (see Text and fonts), documents it produces round-trip cleanly.

Multi-byte codes are handled properly too. A Type 0 font with Identity-H encoding shows two bytes per glyph, and the extractor consumes them in pairs rather than treating each byte as a separate character.

Layout awareness

Content streams contain no spaces between words in the general case — a typesetter emits [(Hello) -250 (world)] TJ and relies on the kerning number to create the gap. Nor is there any newline character; a new line is just a Td/TD/T* operator moving the text position.

The extractor therefore tracks the full text state — the text matrix, the text line matrix, font size, character and word spacing, horizontal scaling, leading and rise — and infers structure from the resulting geometry:

  • A negative adjustment in a TJ array large enough to exceed the font's natural inter-glyph gap becomes a space.
  • A vertical movement of the text position becomes a line break.
  • A large horizontal jump on the same baseline becomes a space, which keeps table columns and multi-column layouts from running together.

The result is text you can read, search and diff — not a perfect reconstruction of the visual layout, which is not recoverable in general, but a faithful one for ordinary flowing documents.

Form XObject recursion

Content is not always in the page's own content stream. A form XObject is a reusable content stream invoked with the Do operator, and it is how repeated elements, imported pages, annotation appearance streams and many third-party writers place their content. A naive extractor that only reads the page stream silently returns nothing for such documents.

Papier recurses into form XObjects when it encounters Do, resolving the XObject through the current resource dictionary and extracting its text with the graphics state in effect at the point of invocation. Nested forms are followed as well, with a depth limit so that a maliciously or accidentally self-referential file cannot loop forever.

Example 29_text_extraction.php builds a document exercising accented Latin text, subset fonts and Type 0 fonts, then reads it back and prints what came out.

Pages and page info

getPages() returns the flattened page list as PdfDictionary objects, in document order, with the page tree already walked and inherited attributes resolved.

foreach ($parser->getPages() as $i => $page) {
    echo 'Page ' . ($i + 1) . ': ' . $parser->extractTextFromPage($page) . "\n";
}

For a structured summary of a single page, use getPageInfo() with a 1-based page number:

$info = $parser->getPageInfo(1);

It returns an array with these keys:

KeyTypeMeaning
numberintThe 1-based page number you asked for.
widthfloatMediaBox width in points.
heightfloatMediaBox height in points.
fontNamesstring[]Resource names of the fonts the page declares.
imageNamesstring[]Resource names of image XObjects on the page.
textstringThe page's extracted text.
annotationsarray[]One entry per annotation, with subtype, rect and contents.

getPageInfo() throws \OutOfRangeException if the page number is below 1 or beyond the page count. Validate user-supplied page numbers against getPageCount() first.

Fonts

getFonts() lists every font referenced anywhere in the document, de-duplicated:

foreach ($parser->getFonts() as $font) {
    printf("%s  %s  %s  %s\n",
        $font['name'], $font['subtype'], $font['baseFont'], $font['encoding']);
}

Each entry has name (the resource name, e.g. F1), subtype (Type1, TrueType, Type0, Type3), baseFont (the /BaseFont value, or '') and encoding (the /Encoding value, or '').

A baseFont with a six-letter uppercase prefix and a plus sign — ABCDEF+Lato-Regular — indicates a subsetted embedded font. That is a useful signal when auditing whether a document is self-contained, since a font without such a prefix and without a descriptor may not be embedded at all.

Images

foreach ($parser->extractImages() as $img) {
    printf("page %d  %s  %dx%d  %s  %d bytes\n",
        $img['page'], $img['name'], $img['width'], $img['height'],
        $img['filter'], strlen($img['data']));

    file_put_contents("img_{$img['page']}_{$img['name']}.bin", $img['data']);
}

Each entry has page (1-based), name (resource name), width, height, filter and data.

data is the result of running the stream's decode chain. For Flate- or LZW-compressed images that means raw uncompressed samples, which you must interpret using the width, height, bit depth and colour space. For DCTDecode, JPXDecode and JBIG2Decode those filters are pass-through, so data is the original JPEG, JPEG 2000 or JBIG2 bitstream — write it out with the matching extension and it opens in any image tool. See Stream filters for the full list.

Metadata

PDF stores document metadata in two places: the legacy /Info dictionary and an XMP packet in the catalog's /Metadata stream. Papier reads both.

$meta = $parser->getMetadata();
// ['title' => …, 'author' => …, 'subject' => …,
//  'creator' => …, 'keywords' => …, 'producer' => …]

echo $meta['title'];

Individual accessors exist for the common fields — getTitle(), getAuthor(), getSubject(), getCreator() — and all of them return '' rather than null when the key is missing, so they are safe to print directly.

Text strings in the Info dictionary may be encoded as PDFDocEncoding or as UTF-16BE with a byte-order mark. Papier detects which and returns UTF-8 either way, so titles with accents or non-Latin scripts come through correctly.

For the XMP packet, which carries the richer RDF/XML metadata used by PDF/A and by digital-asset systems:

$xmp = $parser->getXmpMetadata();   // raw XML string, or null if absent

if ($xmp !== null) {
    $rdf = new SimpleXMLElement($xmp);
}

Annotations

foreach ($parser->getAnnotations() as $a) {
    printf("page %d  %-10s [%s]  %s\n",
        $a['page'], $a['subtype'], implode(', ', $a['rect']), $a['contents']);
}

Each entry has page (1-based), subtype (Text, Link, Highlight, Widget, …), rect as a float array [llx, lly, urx, ury] in default user space, and contents, the annotation's text note.

Note that form fields appear here too, as Widget annotations — a widget is the visual half of an AcroForm field. If you want the field values rather than their rectangles, use getFormFields() instead. See Annotations for the writing side.

Outlines (bookmarks)

getOutlines() returns the bookmark tree, preserving nesting:

function printOutline(array $nodes, int $depth = 0): void
{
    foreach ($nodes as $node) {
        echo str_repeat('  ', $depth) . $node['title'] . "\n";
        printOutline($node['children'], $depth + 1);
    }
}

printOutline($parser->getOutlines());

Each node has title, dest (a string summary of the /Dest target, or '') and children, an array of the same shape. The result is an empty array when the document has no outline. See Navigation for building outlines.

Form fields

foreach ($parser->getFormFields() as $field) {
    printf("%-24s %-4s %s\n", $field['name'], $field['type'], $field['value']);
}

Only terminal (leaf) fields are returned — intermediate nodes in a hierarchical field tree are traversed but not listed, since they hold no value. Each entry has name (the fully-qualified name, with the dot-separated path assembled from the parent chain, e.g. person.name), type (Tx for text, Btn for buttons and checkboxes, Ch for choice fields, Sig for signatures, or '') and value, the /V entry rendered as a string.

To change field values rather than read them, use Papier\AcroForm\FormFiller, described in Forms — it regenerates appearance streams and writes an incremental update.

Attachments

Embedded files live in the catalog's /Names /EmbeddedFiles name tree. Papier flattens the tree for you:

foreach ($parser->getAttachments() as $file) {
    printf("%s (%s, %d bytes)\n", $file['name'], $file['mime'], strlen($file['data']));
    file_put_contents('/tmp/' . basename($file['name']), $file['data']);
}

Each entry has name (the display file name), data (the fully decoded contents) and mime (the /Subtype MIME type, or '').

Attachments are arbitrary bytes chosen by whoever produced the PDF. Treat name as untrusted when writing to disk — run it through basename() and validate it, or a crafted name containing path separators can escape your target directory.

Tagged documents

For accessibility auditing you can check whether a document carries a logical structure tree and reach its root:

if ($parser->isTagged()) {
    $root = $parser->getStructTreeRoot();   // PdfDictionary|null
}

isTagged() reports whether the catalog declares /MarkInfo with /Marked true, which is the prerequisite for PDF/UA and the accessible PDF/A levels. See Accessibility.

Low-level access

When the high-level getters do not cover what you need, you can walk the object graph yourself. References are resolved on demand and cached.

$catalog = $parser->getCatalog();                 // PdfDictionary|null
$obj     = $parser->resolveObject(12, 0);         // by object and generation number
$value   = $parser->resolve($someReference);      // follow a reference one step

$xref    = $parser->getXref();                    // CrossReferenceTable
$raw     = $parser->getRawData();                 // the original bytes
$cache   = $parser->getObjectCache();             // everything resolved so far
$id      = $parser->getFileId();                  // first half of the /ID array

resolve() is the one to reach for inside loops: it returns non-reference objects unchanged, so you can call it unconditionally on any value you pulled out of a dictionary. resolveObject() returns null for object numbers the cross-reference table does not know about, rather than throwing, so a dangling reference degrades gracefully.

The CrossReferenceTable returned by getXref() exposes the entry points you need for incremental updates — getCatalogObjectNumber(), getInfoObjectNumber(), getEncryptObjectNumber(), getSize(), and per-object queries such as getOffset(), isCompressed() and getCompressedLocation().

Robustness against malformed files

Files in the wild are frequently broken in ways the specification does not anticipate. The parser is written defensively throughout, on the principle that a document should yield whatever it can rather than fail wholesale:

  • Broken cross-references. If startxref is missing, points at the wrong place, or the section will not parse, the parser scans the file for N G obj patterns and rebuilds the index. If the trailer has no usable /Root it searches for a dictionary with /Type /Catalog. This happens automatically — there is no flag to set. See recovery.
  • Wrong types. Every getter checks the class of what it resolved before using it. A /Width that is a string instead of an integer yields 0, not a fatal error.
  • Missing keys. Absent dictionary entries produce empty strings and empty arrays rather than warnings.
  • Dangling references. A reference to an object number that does not exist resolves to null and is skipped.
  • Cycles. Page-tree and structure-tree walks are depth-limited, so a self-referential file cannot cause infinite recursion.
  • Mixed structures. Classic tables, cross-reference streams, hybrid-reference files and linearized files are all handled by the same code path.

The consequence worth internalising is that an empty result from a getter means "this document does not have that, or it was unreadable" — never "the parser crashed". Check for empty results rather than catching exceptions.
