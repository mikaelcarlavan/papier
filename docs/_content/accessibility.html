Tagged PDF & PDF/A

A PDF page is a set of drawing instructions. It records that glyphs appear at certain coordinates, but nothing about which glyphs form a heading, which form a paragraph, or what order a human should read them in. Tagged PDF supplies that missing semantic layer, and PDF/A adds the constraints that make a document reliably readable decades from now.

Why tagging matters

Without tags a screen reader has to guess. It infers reading order from the sequence of drawing operators, which frequently differs from the visual order — a two-column layout may be drawn column by column or line by line across the gutter, and the reader has no way to tell which. Tables become a stream of disconnected cells, and images are silent.

Tagging attaches a parallel structure tree to the document: a hierarchy of elements — Document, H1, P, Table, Figure — where each element claims the specific pieces of page content it corresponds to. Assistive technology walks the tree rather than the page, and gets reading order, heading levels and alternative text.

Tagging is also required by law in many jurisdictions for public documents, and it is a hard prerequisite for the accessible PDF/A conformance levels and for PDF/UA.

Marked content and MCIDs

The link between the visible content and the structure tree is the marked-content identifier, or MCID. Content is wrapped in a marked-content sequence carrying a tag and a numeric identifier; a structure element then claims that identifier.

Tagging is a content-stream-level operation, so it uses ContentStream directly:

use Papier\Content\ContentStream;

$cs = new ContentStream();
$cs->setFontResolver(fn (string $n) => $doc->getWriter()->getFontMetrics($n));

$cs->beginMarkedContentMcid('H1', 0)
   ->beginText()
       ->setFont($font, 24)
       ->setTextPosition(72, 760)
       ->showText('Quarterly Report')
   ->endText()
   ->endMarkedContent();

$cs->beginMarkedContentMcid('P', 1)
   ->beginText()
       ->setFont($font, 12)
       ->setTextPosition(72, 720)
       ->showText('This paragraph is tagged so assistive technology can read it.')
   ->endText()
   ->endMarkedContent();

$page->addContent($cs);

beginMarkedContentMcid($tag, $mcid) emits a BDC operator carrying the structure type and the identifier; endMarkedContent() emits the matching EMC.

MCIDs are numbered per page, starting at zero and increasing. Each must be unique within its page, but pages restart the numbering independently.

Every beginMarkedContentMcid() needs a matching endMarkedContent(). Unbalanced markers corrupt the content stream. Sequences may nest, but they may not overlap.

The related beginMarkedContentProps($tag, $propsName) is used for optional content rather than structure — see Layers & multimedia.

The structure tree

With the content marked, build the tree that gives it meaning. StructTreeRoot is the root, and StructElement is a node.

use Papier\LogicalStructure\{StructElement, StructTreeRoot};

$tree = new StructTreeRoot();

$docElem = new StructElement('Document');

$h1 = new StructElement('H1');
$h1->addMCID(0, $page->getDictionary());

$p1 = new StructElement('P');
$p1->addMCID(1, $page->getDictionary());

$p2 = new StructElement('P');
$p2->addMCID(2, $page->getDictionary());

$docElem->addChild($h1)->addChild($p1)->addChild($p2);
$tree->addChild($docElem);

$doc->setStructTree($tree);

addMCID() takes the identifier and the page dictionary the marked content lives on — the page reference is what allows an element to claim content across several pages, as a paragraph spanning a page break does.

Papier derives the rest from this tree. When the document is written it emits /MarkInfo with /Marked true, the /StructTreeRoot in the catalog, a /ParentTree number tree mapping every MCID back to its owning element, and a /StructParents key on each page linking it into that tree. These are the reverse mappings a viewer uses to answer "what does this piece of content mean?" — for example when the user selects text and asks for its structure — and getting them consistent by hand is tedious and error-prone.

Structure types

The type string passed to StructElement should come from the standard structure namespace. The commonly useful ones:

TypeMeaning
DocumentThe root of a complete document
Part, Sect, DivGeneric grouping levels
H1–H6Headings, in descending significance
PA paragraph
L, LI, Lbl, LBodyList, list item, item label, item body
Table, TR, TH, TDTable, row, header cell, data cell
FigureAn image or illustration
CaptionA caption for a figure or table
LinkA hyperlink
SpanAn inline run with distinct properties
ArtifactDecoration with no semantic content

Use headings in proper order — do not skip from H1 to H3 because it looks right, since screen-reader users navigate by heading level. And mark purely decorative content as Artifact so it is skipped rather than read aloud.

Alternative text and language

The StructElement constructor takes three optional arguments after the type:

$figure = new StructElement(
    'Figure',
    lang:       'en-GB',
    altText:    'Bar chart showing quarterly revenue rising from £1.2M to £3.4M.',
    actualText: null,
);
$figure->addMCID(3, $page->getDictionary());

altText becomes /Alt — the description read in place of content that cannot be read directly. Every Figure needs one; a chart or diagram with no alternative text is simply absent for a screen-reader user.

actualText becomes /ActualText and is subtly different: it is the text this content is, for cases where the glyphs do not spell it. A decorative drop cap drawn as a graphic, or a ligature encoded as a single glyph, uses /ActualText so that extraction and search still work.

lang sets the natural language of the subtree, which lets a screen reader switch pronunciation for a quoted passage in another language.

Two further methods: setTitle() writes /T, a human-readable title for the element, and setAttribute() attaches an attribute dictionary — that is how table cells declare /ColSpan and /RowSpan, and how a list declares its numbering style.

Tagging tables

Tables are where tagging pays off most, and where it is most often skipped. An untagged table is read as an undifferentiated stream of cell contents with no indication of which header each value belongs to.

$table = new StructElement('Table');

$headerRow = new StructElement('TR');
foreach ([0, 1, 2] as $mcid) {
    $th = new StructElement('TH');
    $th->addMCID($mcid, $page->getDictionary());
    $headerRow->addChild($th);
}
$table->addChild($headerRow);

$dataRow = new StructElement('TR');
foreach ([3, 4, 5] as $mcid) {
    $td = new StructElement('TD');
    $td->addMCID($mcid, $page->getDictionary());
    $dataRow->addChild($td);
}
$table->addChild($dataRow);

$docElem->addChild($table);

Each cell's content must be wrapped in its own marked-content sequence with the matching MCID. See Tables for drawing the table itself.

PDF/A archival conformance

PDF/A (ISO 19005) is a profile for long-term preservation. It rests on one principle: the document must be self-contained. Everything needed to render it exactly as intended has to be inside the file, because in twenty years the fonts, colour profiles and external resources it depends on may no longer exist.

That principle produces the constraints. Fonts must be embedded. Colour must be unambiguous, via an embedded output intent. Encryption is forbidden, since an archive must not depend on a key. So are JavaScript, external references, and embedded content in non-conforming formats.

Enabling PDF/A

use Papier\Metadata\PdfAConformance;

$doc->enablePdfA(2, PdfAConformance::Basic);   // PDF/A-2b

This adds an sRGB OutputIntent with an embedded ICC profile, PDF/A identification metadata in the XMP packet, and a document ID in the trailer.

The first argument is the PDF/A part: 1 (based on PDF 1.4, the most restrictive), 2 (based on PDF 1.7 — the default and the usual choice), or 3 (PDF/A-2 plus arbitrary embedded attachments).

The second is the conformance level:

CaseLevelRequirement
PdfAConformance::BasicBReliable visual reproduction only
PdfAConformance::AccessibleAB, plus full Tagged PDF and Unicode mapping
PdfAConformance::UnicodeUB, plus Unicode mapping for all text (PDF/A-2 and later)

Level A is the accessible one, and it is where this page's two halves meet: it requires everything level B requires and a complete structure tree.

$doc->enablePdfA(2, PdfAConformance::Accessible);   // PDF/A-2a
// … then build and set a structure tree, as above

Embedding fonts

The most common conformance failure is an unembedded font. The 14 standard Type 1 fonts are not embedded — a viewer is expected to supply them — so they are not permitted in PDF/A. Always load fonts from a file:

// Wrong for PDF/A — relies on a viewer-supplied font
$font = $doc->addFont('Helvetica');

// Correct — the font program is embedded in the file
$font = $doc->addFont(__DIR__ . '/Lato-Regular.ttf', '', subset: true);

Subsetting is safe and recommended: only the glyphs actually used are embedded, and Papier generates a /ToUnicode CMap so the text remains searchable and copy-pasteable — which is itself a level U and level A requirement. For non-Latin scripts use addUnicodeFont(). See Text & fonts.

Check that the font's licence permits embedding. Some commercial fonts forbid it, and some allow only preview-and-print embedding.

A complete PDF/A document

use Papier\Elements\{Color, Text};
use Papier\Metadata\PdfAConformance;
use Papier\PdfDocument;

$doc = PdfDocument::create();
$doc->setTitle('Annual Report 2025')
    ->setAuthor('Papier')
    ->setSubject('Archival copy');

$doc->enablePdfA(2, PdfAConformance::Basic);

$font = $doc->addFont(__DIR__ . '/Lato-Regular.ttf', '', subset: true);

$page = $doc->addPage();
$page->add(
    Text::write('Annual Report 2025')->at(72, 770)->font($font, 22)->color(Color::hex('#1a1a2e')),
    Text::write('This document is suitable for long-term archiving (PDF/A-2b).')
        ->at(72, 735)->font($font, 12),
);

$doc->save('archive.pdf');

Validating

PdfAValidator checks the structural requirements and returns a list of human-readable problems. An empty array means no issues were found:

use Papier\Validation\PdfAValidator;

$issues = PdfAValidator::validate(file_get_contents('archive.pdf'));

if ($issues === []) {
    echo "Structurally conformant.\n";
} else {
    foreach ($issues as $issue) {
        echo "  - $issue\n";
    }
}

A boolean shortcut is available:

if (PdfAValidator::isValid($pdf)) {
    // …
}

The validator checks that the document parses; that it is not encrypted; that the trailer carries a document /ID; that a GTS_PDFA1 OutputIntent exists with an embedded /DestOutputProfile; that an uncompressed XMP /Metadata stream is present and declares pdfaid; that every font is embedded; and — when the XMP declares conformance level A — that the document is actually tagged.

This is a structural check, not a full conformance test. It cannot verify colour space usage, transparency restrictions, or the semantic quality of a structure tree. Run a complete validator such as veraPDF before treating a document as certified.

Verifying the tagging

The parser exposes the read side, which is the quickest way to confirm the writer emitted what you expected:

use Papier\Parser\PdfParser;

$parser = new PdfParser(file_get_contents('tagged.pdf'));
$parser->parse();

echo $parser->isTagged() ? "tagged\n" : "not tagged\n";

$root = $parser->getStructTreeRoot();
echo $root !== null ? "has StructTreeRoot\n" : "no structure tree\n";
echo $root && $root->get('ParentTree') !== null ? "has ParentTree\n" : "no ParentTree\n";

echo $parser->getXmpMetadata();

isTagged() reports whether /MarkInfo declares /Marked true. More reading recipes are in Reading PDFs.

PDF/UA

PDF/UA (ISO 14289) is the accessibility standard proper. Where PDF/A-2a requires a structure tree to exist, PDF/UA specifies what a correct one looks like — and much of it cannot be verified mechanically.

Building on the tagging described above, a PDF/UA document additionally needs:

  • A structure tree covering all real content, with decoration marked as Artifact.
  • Alternative text on every Figure and every non-text element that conveys meaning.
  • Headings used in proper hierarchical order, without skipped levels.
  • A document language set, and per-element languages for passages that differ.
  • /DisplayDocTitle enabled in viewer preferences, so the window title shows the document title rather than the filename (see Documents & pages).
  • Tooltips (/TU) on interactive form fields — see Forms.
  • Tables tagged with genuine header cells and correct scope.
  • Colour never used as the sole carrier of information, and adequate contrast.

Papier gives you the mechanisms for all of these. Whether the result is genuinely accessible is a matter of authoring judgement, and is worth checking with a real screen reader rather than a validator alone.
