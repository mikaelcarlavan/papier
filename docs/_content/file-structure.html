File structure

A PDF is not a linear document format — it is a small object database with an index at the end. Understanding that layout explains why PDFs can be updated without being rewritten, why a single corrupt byte can make a file unopenable, and why Papier offers several ways to write the same content.

Anatomy of a PDF file

Every PDF written by Papier has four parts, in this order:

  1. Header — a version comment such as %PDF-1.7, followed by a line of high-bit binary bytes so that naive file-transfer tools treat the file as binary rather than text.
  2. Body — a sequence of numbered indirect objects. Each one looks like 12 0 obj … endobj, where 12 is the object number and 0 the generation number. Objects are dictionaries, arrays, numbers, strings, names, or streams; they reference one another with 12 0 R.
  3. Cross-reference section — the index that maps each object number to its byte offset in the file.
  4. Trailer — a dictionary naming the entry points (/Root, /Info, /Encrypt, /ID, /Size), then startxref with the byte offset of the cross-reference section, then %%EOF.

A reader opens a PDF backwards: it seeks to the end, reads startxref, jumps to the cross-reference section, reads the trailer, resolves /Root to the catalog, and walks down to the page tree. Nothing before /Root needs to be read at all, which is what makes random access to page 400 of a 500-page file cheap.

Papier builds this structure for you. You only need to care about it when you want smaller files (object streams), want to edit a file you did not author (incremental updates), or are handed a file that will not open (recovery).

The document catalog

The trailer's /Root points at the catalog, the root of the object graph. From there hang the page tree (/Pages), the outline tree (/Outlines), the interactive form (/AcroForm), the name tree that holds attachments and named destinations (/Names), optional content properties (/OCProperties), the structure tree used for accessibility, and the XMP metadata stream (/Metadata).

When reading, you can reach the catalog directly:

use Papier\Parser\PdfParser;

$parser = new PdfParser(file_get_contents('document.pdf'));
$parser->parse();

$catalog = $parser->getCatalog();   // PdfDictionary|null
$info    = $parser->getInfo();      // the /Info dictionary
$version = $parser->getVersion();   // e.g. '1.7'

Classic cross-reference tables

The original PDF 1.0 index is a plain-text table. It starts with the keyword xref, then one or more subsections. Each subsection declares a starting object number and a count, followed by one fixed-width 20-byte entry per object: a 10-digit byte offset, a 5-digit generation number, and either n (in use) or f (free).

The fixed width is the point — a reader can binary-search or index into the table without parsing it. The cost is size: 20 bytes per object, uncompressed, plus a trailer dictionary that cannot be compressed either. For a document with thousands of small objects that adds up.

Papier writes a classic table by default, because it is readable by every PDF viewer ever shipped.

Cross-reference streams

PDF 1.5 replaced the text table with a cross-reference stream: an ordinary stream object whose /Type is /XRef and whose payload is a binary table, normally Flate-compressed with a predictor. The trailer keys move into the stream's own dictionary, so the trailer keyword disappears entirely and startxref points at the stream object.

Each entry has three fields whose widths are declared by /W: an entry type, and two type-dependent values. Type 1 entries are the familiar offset/generation pairs. Type 2 entries are new — they say "this object lives inside object stream N, at index i", which is what makes object streams possible.

Papier's parser reads classic tables, cross-reference streams, and hybrid-reference files (which carry both, via /XRefStm, so that old and new readers both work). Linearized ("fast web view") files are handled too — their first-page-first layout is just a particular arrangement of the same structures.

Object streams

An object stream (/Type /ObjStm) packs many non-stream objects into the payload of a single stream, which is then compressed as a unit. Instead of a hundred separately-written dictionaries with a hundred uncompressed N 0 obj headers, you get one compressed blob. Because Flate works better on more data, packing similar dictionaries together compresses far better than compressing each one alone.

Streams themselves (content streams, images, fonts) cannot go inside an object stream — they are already binary and already filtered — so they stay in the body. Neither can the encryption dictionary, since it must be readable before decryption starts.

Turn it on with one call before saving:

use Papier\PdfDocument;

$doc = PdfDocument::create();
// … build the document …

$doc->useObjectStreams();       // pack objects into /ObjStm + write an /XRef stream
$doc->save('small.pdf');

useObjectStreams() implies a cross-reference stream, because type 2 xref entries are the only way to point into an object stream. It takes an optional boolean so you can toggle it from a config flag:

$doc->useObjectStreams($config['compressPdf']);

Example 17_compressed_objects.php builds the same eight-page report twice, once classic and once compressed, and prints the size difference. The saving is largest on documents with many small objects — long outline trees, many annotations, many pages.

Object streams require PDF 1.5. Every current viewer supports them, but if you must target a very old reader, or you are producing PDF/A-1 (which forbids them), leave the default classic layout in place.

Incremental updates

An incremental update appends a new revision to the end of an existing file instead of rewriting it. The original bytes stay byte-for-byte identical; after them come only the objects that changed or were added, a new cross-reference section covering just those objects, and a new trailer whose /Prev key points back at the previous cross-reference section. A reader follows the /Prev chain backwards and the newest definition of each object wins.

This matters for two reasons. First, it is cheap — updating one dictionary in a 50 MB file appends a few hundred bytes. Second, and more importantly, it is what makes digital signatures work: a signature covers a byte range of the file, so any later change must be appended rather than woven in, or every earlier signature breaks.

use Papier\PdfDocument;
use Papier\Metadata\DocumentInfo;
use Papier\Objects\{PdfDictionary, PdfName, PdfString};

$updater = PdfDocument::openForUpdate('in.pdf');   // returns an IncrementalUpdater

$info = new DocumentInfo();
$info->setTitle('Draft v2')->setAuthor('Papier');
$updater->updateObject($infoNum, $info->getDictionary());   // override an existing object

$marker = new PdfDictionary();
$marker->set('Type', new PdfName('RevisionMarker'));
$marker->set('Note', new PdfString('Added without rewriting the file.'));
$newNum = $updater->addObject($marker);                     // append a new object

$updater->save('out.pdf');

To override an object you need its number. Get it from the parser's cross-reference table:

use Papier\Parser\PdfParser;

$parser = new PdfParser(file_get_contents('in.pdf'));
$parser->parse();

$xref       = $parser->getXref();
$infoNum    = $xref->getInfoObjectNumber();
$catalogNum = $xref->getCatalogObjectNumber();

The IncrementalUpdater API

  • updateObject(int $objNum, PdfObject $obj) — replace the definition of an existing object in the appended revision. Returns $this.
  • addObject(PdfObject $obj): int — append a brand-new object and return its assigned number.
  • peekNextObjectNumber(): int — the number addObject() would hand out next, useful when two objects must reference each other.
  • build(): string — produce the complete updated file as a string.
  • save(string $path): void — write it to disk.

openForUpdate() takes an optional second argument, the password, so encrypted files can be updated as well:

$updater = PdfDocument::openForUpdate('protected.pdf', 'secret');

You can also construct the updater yourself from a parser you already have, which avoids parsing the file twice:

use Papier\Writer\IncrementalUpdater;

$updater = new IncrementalUpdater($parser);

Example 18_incremental_update.php demonstrates the whole cycle and then verifies that the original bytes are still a literal prefix of the updated file. The same mechanism backs FormFiller::save() and PdfSigner::sign().

Recovering damaged files

Because everything hangs off startxref, a PDF is unusually fragile at its tail. Truncated downloads, naive text-mode transfers, and buggy third-party writers all produce files whose startxref offset is wrong, whose cross-reference table is missing, or whose trailer never made it to disk.

Papier's parser handles this without a special flag. If the offset does not land on a cross-reference section, or the section fails to parse, or the trailer has no /Root, it falls back to scanning the raw bytes for every N G obj pattern, rebuilding the offset table from what it finds, and then locating the catalog by looking for a dictionary with /Type /Catalog. Later definitions of the same object number win, which mirrors how revisions layer in a healthy file.

use Papier\Parser\PdfParser;

$parser = new PdfParser($maybeBrokenBytes);
$parser->parse();               // rebuilds the xref automatically if needed

echo $parser->getPageCount();
echo $parser->extractText();

Example 25_xref_recovery.php takes a valid document, deliberately breaks it two ways — corrupting the startxref offset, then stripping the cross-reference section and trailer entirely — and reads both back successfully.

Recovery is best-effort. It can rebuild the index, but it cannot invent objects that were truncated away. If the tail of the body is gone, the pages that lived there are gone too.

Stream filters

Stream objects declare a /Filter — one name, or an array of names applied in order — plus optional /DecodeParms. Papier implements the filters from ISO 32000-1 §7.4 behind a single factory, so encoding and decoding are symmetric calls.

Available filters

NameAbbreviationPurpose
FlateDecodeFlzlib/deflate. The default for content streams, fonts and metadata. Supports PNG and TIFF predictors.
LZWDecodeLZWLegacy LZW compression from PDF 1.0–1.1. Read for compatibility; also supports predictors.
ASCIIHexDecodeAHxBinary as hex digits, terminated by >. Doubles the size; used when a stream must be 7-bit clean.
ASCII85DecodeA85Base-85 ASCII encoding. ~25% overhead instead of 100%.
RunLengthDecodeRLSimple run-length coding. Cheap and effective on flat-colour bitmaps.
CCITTFaxDecodeCCFGroup 4 (T.6) bilevel compression for scanned black-and-white pages. Encode and decode.
DCTDecodeDCTBaseline JPEG. Pass-through — the JPEG bytes are embedded and extracted unchanged.
JBIG2Decode—JBIG2 bilevel images. Pass-through.
JPXDecode—JPEG 2000 images. Pass-through.
CryptIdentityPer-stream encryption selector used by the security handler.

"Pass-through" means Papier stores and retrieves the compressed bytes without transcoding them. That is the correct behaviour for image codecs: the viewer's own decoder is faster and better than anything a PHP implementation could do, and re-encoding would lose quality. When you call extractImages() you get the JPEG or JPEG 2000 bytes, which you can hand straight to an image library or write to a file.

Using filters directly

use Papier\Filter\FilterFactory;

$compressed = FilterFactory::encode('FlateDecode', $rawBytes);
$roundTrip  = FilterFactory::decode('FlateDecode', $compressed);

$filter = FilterFactory::create('ASCII85Decode');   // a FilterInterface instance

You can register your own filter for a non-standard name; it only has to implement Papier\Filter\FilterInterface:

FilterFactory::register('MyCustomDecode', MyCustomFilter::class);

Predictors

A predictor is a reversible transform applied before compression that makes data more compressible. Instead of storing each byte, you store its difference from a neighbour — in image data neighbours are usually similar, so the differences cluster near zero and Flate encodes them in far fewer bits.

Papier's FlateDecode and LZWDecode honour the /Predictor parameter in /DecodeParms, together with /Columns, /Colors and /BitsPerComponent. Predictor 2 is the TIFF horizontal differencing scheme; predictors 10–15 are the PNG filters (None, Sub, Up, Average, Paeth, and Optimum, which chooses per row). Cross-reference streams almost always use predictor 12, because their entries are near-identical row to row.

CCITT Group 4

Group 4 is the encoding behind fax machines and scanned document archives. It codes a bilevel image by describing each row relative to the one above it, which is extremely effective on text scans — an order of magnitude smaller than Flate on the same data is typical.

use Papier\Filter\CCITTFaxDecode;
use Papier\Objects\{PdfDictionary, PdfInteger};

$params = new PdfDictionary();
$params->set('K', new PdfInteger(-1));          // K < 0 selects Group 4 (T.6)
$params->set('Columns', new PdfInteger($cols));
$params->set('Rows', new PdfInteger($rows));

$codec   = new CCITTFaxDecode();
$encoded = $codec->encode($bilevelBytes, $params);
$decoded = $codec->decode($encoded, $params);   // also reads scanned third-party PDFs

The recognised /DecodeParms keys are Columns (default 1728), Rows, K, BlackIs1 (default false) and EncodedByteAlign (default false). Group 3 encodings (K >= 0) are not decoded.

To embed the result, build an image XObject whose /Filter is /CCITTFaxDecode, with /BitsPerComponent 1 and /ColorSpace /DeviceGray, and attach the same parameter dictionary as /DecodeParms. Example 31_ccitt_fax.php shows the full construction.
