Encryption & signatures

Encryption and digital signatures answer two different questions. Encryption controls who can open a document and what they may do with it. A signature proves who produced it and that nothing has changed since. They are independent, and most workflows need only one.

Encrypting a document

One call on the document sets up the standard security handler:

use Papier\Encryption\{EncryptionAlgorithm, StandardSecurityHandler};

$doc->encrypt(
    userPassword:  'user',
    ownerPassword: 'owner',
    permissions:   StandardSecurityHandler::PERM_PRINT | StandardSecurityHandler::PERM_COPY,
    algorithm:     EncryptionAlgorithm::Aes_256,
);

$doc->save('protected.pdf');

Call encrypt() before save(). Every string and stream in the file is encrypted as it is written.

The two passwords

The standard security handler defines two roles, which is the part people most often get wrong.

The user password is required to open the document. Leave it empty and anyone can open the file — but the permission restrictions still apply, which is the usual configuration for a document that should be readable but not printable or copyable.

The owner password bypasses every restriction. Someone who knows it gets full access regardless of the permission flags. If it is empty it defaults to the user password, which means the restrictions are trivially removable — so always set a distinct owner password when the permissions matter.

// Anyone can open; only the owner can change the restrictions
$doc->encrypt('', 'owner-secret', StandardSecurityHandler::PERM_PRINT);

// A password is needed to open at all
$doc->encrypt('open-sesame', 'owner-secret');

Permission flags are advisory. They are enforced by cooperating viewers, not by cryptography — a reader holding the user password holds the decryption key, and any tool that ignores the flags can print or copy freely. Use permissions to express intent, not to enforce confidentiality. Only the user password provides real protection.

Permission flags

Permissions are a bit field. OR the constants together to build the set you want:

ConstantGrants
PERM_PRINTPrinting (at degraded resolution unless PERM_PRINT_HQ is also set)
PERM_MODIFYChanging the document's contents
PERM_COPYCopying text and graphics to the clipboard
PERM_ANNOTAdding or modifying annotations and form fields
PERM_FILL_FORMFilling in existing form fields
PERM_EXTRACTExtracting content for accessibility purposes
PERM_ASSEMBLEInserting, rotating or deleting pages, and adding bookmarks
PERM_PRINT_HQPrinting at full resolution
PERM_ALLAll of the above — the default
 
// Read and print, but no copying and no editing
$permissions = StandardSecurityHandler::PERM_PRINT
             | StandardSecurityHandler::PERM_PRINT_HQ
             | StandardSecurityHandler::PERM_FILL_FORM
             | StandardSecurityHandler::PERM_EXTRACT;

$doc->encrypt('user', 'owner', $permissions, EncryptionAlgorithm::Aes_128);

Leave PERM_EXTRACT set unless you have a strong reason not to. It is the flag screen readers rely on, and clearing it makes the document inaccessible to assistive technology. See Tagged PDF & PDF/A.

Choosing an algorithm

The EncryptionAlgorithm enum offers four options spanning twenty years of the specification:

CaseCipherPDF versionRecommendation
Rc4_40RC4, 40-bit key1.1Broken. Legacy compatibility only.
Rc4_128RC4, 128-bit key1.4Weak cipher. Avoid for new documents.
Aes_128AES-1281.6The default. Sound, and universally supported.
Aes_256AES-2562.0Strongest. Needs a reasonably modern viewer.
 
$doc->encrypt('user', 'owner', StandardSecurityHandler::PERM_ALL, EncryptionAlgorithm::Aes_256);

Aes_128 is the default because it is both secure and supported everywhere. Choose Aes_256 when you control the audience's tooling — it also uses a stronger password-derivation scheme, which matters if the passwords are short.

Encryption strength is bounded by password strength. Any of these algorithms falls quickly to a dictionary attack against a weak password. Use long, random passwords for anything genuinely sensitive.

Using the handler directly

StandardSecurityHandler is what encrypt() constructs internally. You rarely need it directly, but it is available for low-level work — buildEncryptDictionary() produces the /Encrypt dictionary, getEncryptionKey() exposes the derived file key, and encryptData() encrypts a single object's bytes.

Reading encrypted PDFs

The parser decrypts transparently once a password is supplied. Set it before calling parse():

use Papier\Parser\PdfParser;

$parser = (new PdfParser(file_get_contents('protected.pdf')))
    ->setPassword('open-sesame');
$parser->parse();

echo $parser->isEncrypted() ? "encrypted\n" : "not encrypted\n";
echo $parser->getTitle() . "\n";
echo $parser->extractText() . "\n";

Either password works — the parser detects which one it was given and derives the file key accordingly:

$asUser  = (new PdfParser($pdf))->setPassword('open-sesame');
$asOwner = (new PdfParser($pdf))->setPassword('owner-key');

Once decrypted, every read-side feature works normally — outlines, form fields, attachments, XMP:

$parser->parse();

foreach ($parser->getOutlines() as $item) {
    echo "• {$item['title']}\n";
}
foreach ($parser->getFormFields() as $f) {
    echo "  {$f['name']} ({$f['type']}) = {$f['value']}\n";
}
foreach ($parser->getAttachments() as $a) {
    echo "  {$a['name']} [{$a['mime']}], " . strlen($a['data']) . " bytes\n";
}
echo $parser->getXmpMetadata();

Documents with an empty user password need no setPassword() call at all. RC4-40, RC4-128, AES-128 and AES-256 are all handled; StandardDecryptor does the work internally.

Digital signatures

A digital signature binds a document's bytes to a certificate. Papier produces PKCS#7 (CMS) detached signatures and adds them as an incremental update: the original bytes are preserved verbatim and a new revision is appended. That is what allows several signatures to coexist, each one validating the state of the document at the moment it was applied.

Signing a document

use Papier\Signature\PdfSigner;

$signer = new PdfSigner($certPem, $keyPem);   // PEM strings or file paths
$signer->setName('Alice Example')
       ->setReason('I approve this agreement')
       ->setLocation('Paris')
       ->setContactInfo('alice@example.com');

$signed = $signer->sign(file_get_contents('agreement.pdf'));
file_put_contents('signed.pdf', $signed);

sign() takes and returns PDF bytes, so it composes naturally with toString():

$unsigned = $doc->toString();
$signed   = $signer->sign($unsigned);

// The original is preserved as a literal prefix of the signed file
var_dump(str_starts_with($signed, $unsigned));   // true

That prefix property is the whole point of incremental signing: a verifier can reconstruct exactly the bytes that were signed.

Additional options: setDate('YYYYMMDDHHMMSS') pins the claimed signing time, addCertificate($pem) embeds intermediate CA certificates so verifiers can build the trust chain, and setSignatureCapacity($bytes) enlarges the reserved /Contents placeholder when a long certificate chain will not fit the default.

Obtaining a certificate

Production signing needs a certificate from a CA your recipients trust. For testing, generate a throwaway self-signed pair with the OpenSSL extension:

$pkey = openssl_pkey_new(['private_key_bits' => 2048, 'private_key_type' => OPENSSL_KEYTYPE_RSA]);
$csr  = openssl_csr_new(
    ['commonName' => 'Papier Demo', 'organizationName' => 'Papier'],
    $pkey,
    ['digest_alg' => 'sha256'],
);
$x509 = openssl_csr_sign($csr, null, $pkey, 365, ['digest_alg' => 'sha256']);

openssl_x509_export($x509, $certPem);
openssl_pkey_export($pkey, $keyPem);

A self-signed certificate produces a cryptographically valid signature that viewers will still flag as untrusted, because nothing vouches for the identity. Use it for development only.

Visible signatures

By default the signature is invisible — present and verifiable, but with nothing drawn on the page. setVisibleAppearance() adds a signature box showing the name, reason, location and date:

$signer = (new PdfSigner($certPem, $keyPem))
    ->setName('Alice Example')
    ->setReason('I approve this agreement')
    ->setLocation('Paris')
    ->setVisibleAppearance(x: 360, y: 690, w: 170, h: 60, page: 1);

$signed = $signer->sign($unsigned);

The coordinates are in points from the bottom-left of the page, and page is 1-based. The appearance is a form XObject attached to a signature widget annotation.

Whether a signature is visible has no bearing on its cryptographic validity — a visible box is a convenience for human readers, and an invisible signature is equally binding.

Multiple signatures

Because each signature is an appended revision, signing an already-signed document is just another call:

$signedOnce  = (new PdfSigner($aliceCert, $aliceKey))->setName('Alice')->sign($pdf);
$signedTwice = (new PdfSigner($bobCert,   $bobKey))->setName('Bob')->sign($signedOnce);

Alice's signature still validates against the revision she signed, and Bob's covers the whole file including her signature. Each %%EOF marker in the output marks one revision.

If you plan to reserve a signature slot in the document itself rather than signing after the fact, add a SignatureField — see Forms.

Document timestamps

A signature asserts a signing time, but that time comes from the signer's own clock and can be set to anything. A document timestamp fixes the problem by having a trusted third party — a Time Stamping Authority — attest that the document existed at a given moment. This is the /DocTimeStamp mechanism of RFC 3161, and it is what raises a plain signature to PAdES baseline conformance.

DocumentTimestamp takes a closure that performs the TSA round trip. Papier deliberately does not include an HTTP client: you supply the transport, so the class works with any TSA and any HTTP library.

use Papier\Signature\DocumentTimestamp;

$ts = new DocumentTimestamp(function (string $digest): string {
    $request  = DocumentTimestamp::buildRequest($digest);   // RFC 3161 request, DER encoded

    $ch = curl_init('https://tsa.example.com/tsr');
    curl_setopt_array($ch, [
        CURLOPT_POST           => true,
        CURLOPT_POSTFIELDS     => $request,
        CURLOPT_HTTPHEADER     => ['Content-Type: application/timestamp-query'],
        CURLOPT_RETURNTRANSFER => true,
    ]);
    $response = curl_exec($ch);
    curl_close($ch);

    return DocumentTimestamp::extractToken($response);      // pull the token out
});

$timestamped = $ts->apply($signed);
file_put_contents('timestamped.pdf', $timestamped);

The closure receives the message digest to be timestamped and must return the DER-encoded timestamp token. The two static helpers handle the protocol details: buildRequest() builds a valid RFC 3161 query (its second argument, $certReq, asks the TSA to include its certificate), and extractToken() pulls the token out of the response.

setCapacity() enlarges the reserved space if your TSA returns an unusually large token.

Like signing, apply() appends a revision, so a timestamp can be added to an already-signed document without invalidating the signature.

Combining signing and encryption

The two features interact awkwardly, and the order matters. Encrypt first and then sign: the signature then covers the encrypted bytes as they will actually be distributed. Signing first and encrypting afterwards rewrites the whole file and breaks the signature.

PDF/A forbids encryption outright — an archival document must remain readable without a key. Signatures are permitted. See Tagged PDF & PDF/A.
