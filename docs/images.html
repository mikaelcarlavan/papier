Images

The Image element loads JPEG and PNG data, embeds it as an image XObject, and places it on the page. Transparency, colour space and bit depth are worked out from the file, so the common case is a single chained expression.

Loading an image

There are three constructors. fromFile() reads a path and sniffs the format; the other two take raw bytes when you already have the data in memory — from a database, an upload, or an image you generated with GD.

use Papier\Elements\Image;

Image::fromFile(__DIR__ . '/photo.png');      // JPEG or PNG, auto-detected
Image::fromJpeg(file_get_contents('photo.jpg'));
Image::fromPng($pngBytes);

Nothing is drawn until you add the element to a page:

$page->add(
    Image::fromFile(__DIR__ . '/photo.png')->at(72, 500)->fitWidth(300),
);

JPEG data is passed straight through to the PDF with a /DCTDecode filter — the bytes are copied verbatim, with no decode and re-encode, so there is no generation loss and no CPU cost. PNG is decoded, then re-compressed with Flate, which is the closest PDF equivalent.

Positioning

at() sets the bottom-left corner of the placed image, matching the PDF coordinate system where y grows upward. To place an image so its top edge sits at a known y, subtract the height you are about to give it.

$page->add(Image::fromFile($path)->at(72, 640)->size(150, 100));

Sizing

An image element must be told how large to draw. Three methods cover the cases, and the last one called wins.

Explicit size

Image::fromJpeg($bytes)->at(72, 640)->size(150, 100);

size() takes width and height in points and does not preserve aspect ratio. Use it when the target box is fixed and you know the source already matches it, or when distortion is intentional.

Proportional fitting

Image::fromJpeg($wide)->at(72, 240)->fitWidth(160);
Image::fromJpeg($tall)->at(250, 200)->fitHeight(80);

fitWidth() sets the width and derives the height from the source aspect ratio; fitHeight() does the reverse. These are what you want almost all the time — laying out a column of images at a fixed width, or a row of thumbnails at a fixed height, without having to know or compute each source's proportions.

MethodBehaviour
size($w, $h)Exact dimensions; aspect ratio not preserved
fitWidth($w)Width fixed, height derived from the source
fitHeight($h)Height fixed, width derived from the source

Points are not pixels. A 600 × 400 pixel photograph drawn at fitWidth(300) occupies 300 pt — about 4.2 inches — and therefore prints at roughly 144 dpi. To hit 300 dpi, draw at pixel width divided by 300 times 72; for the same image that is 144 pt.

Opacity

Image::fromJpeg($bytes)->at(350, 240)->size(140, 42)->opacity(0.4);

opacity() takes a value from 0 to 1 and is applied through an extended graphics state, so it composites with whatever is already on the page. It is independent of any alpha channel in the image itself; a translucent PNG drawn at opacity(0.5) ends up half as opaque again.

The obvious use is a background watermark image behind body text.

PNG alpha and soft masks

PDF has no alpha channel on image XObjects. Transparency is expressed instead through a separate greyscale image, referenced from the main image's /SMask entry, whose grey values are read as opacity.

Papier does this for you. When fromPng() or fromFile() encounters a PNG with an alpha channel, it splits the alpha out into its own stream, embeds it as an SMask, and links the two. Nothing in your code changes:

use Papier\Elements\{Color, Image, Rectangle};

$page->add(
    Rectangle::create(72, 490, 200, 100)->fill(Color::rgb(0.9, 0.9, 0.2)),
    Image::fromPng($transparentPng)->at(72, 490)->size(200, 100),
);

The yellow rectangle shows through wherever the PNG is transparent, with full 8-bit gradation — not just a binary cut-out. This handles anti-aliased edges and soft shadows correctly, which is why PNG is the right format for logos and overlays and JPEG is not.

At the lower level, PngImage exposes hasAlpha() and getSMaskStream() if you need to inspect or reuse the mask, and any PdfImage accepts an explicit setSoftMask().

Image masks and stencils

A stencil mask is a 1-bit image that does not carry colour at all. Instead it selects which pixels of the fill colour get painted, so the same mask can be stamped in any colour you like — the usual way to place a monochrome logo or a signature scan.

use Papier\Graphics\Image\PdfImage;

$mask = new PngImage($bilevelPngBytes);
$mask->setImageMask(true)
     ->setBitsPerComponent(1)
     ->setDecode(new PdfArray([new PdfInteger(0), new PdfInteger(1)]));

$page->getResources()->addXObject('Stencil', $mask->getStream());

$cs->save()
   ->setFillRGB(0.75, 0.1, 0.1)
   ->transform(120, 0, 0, 60, 72, 500)   // width, 0, 0, height, x, y
   ->drawXObject('Stencil')
   ->restore();

Two things to notice. First, an image mask carries no colour space, so it takes the current fill colour — set it before drawing. Second, image XObjects are always drawn into the unit square, so you scale and position them with a transformation matrix rather than with coordinates. The six numbers above mean "120 wide, 60 tall, bottom-left at (72, 500)".

Decode arrays

The /Decode array remaps each sample value's range before the image is interpreted. Supplying [1, 0] for a single-component image inverts it — what was black becomes white.

$img->setDecode(new PdfArray([new PdfInteger(1), new PdfInteger(0)]));   // invert

The array has two entries per component, giving the output value that the minimum and maximum sample values map to. For an image mask, the decode array decides which bit value paints and which is transparent, which is how you flip a stencil that came out inverted. Inverting through /Decode costs nothing, since the reader applies it during decode rather than requiring you to rewrite the pixel data.

Lower-level image control

Behind the element sit JpegImage and PngImage, both extending PdfImage. Use them directly when you need to set entries the element does not expose.

use Papier\Graphics\Image\JpegImage;

$img = JpegImage::fromFile(__DIR__ . '/photo.jpg');
$img->setInterpolate(true);

echo $img->getWidth() . '×' . $img->getHeight();

$page->getResources()->addXObject('Im1', $img->getStream());
$cs->save()->transform(300, 0, 0, 200, 72, 400)->drawXObject('Im1')->restore();
MethodPurpose
getWidth() / getHeight()Source dimensions in pixels
setColorSpace(string $cs)Override the detected colour space
setBitsPerComponent(int $bpc)1, 2, 4, 8 or 16
setInterpolate(bool $on)Smooth the image when scaled up
setImageMask(bool $on)Treat as a 1-bit stencil
setSoftMask(PdfObject $mask)Attach an alpha mask
setDecode(PdfObject $decode)Sample value remapping
getStream()The image XObject stream

setInterpolate(true) asks the reader to smooth the image when it is scaled beyond its native resolution. It helps with a low-resolution photograph and hurts with a deliberately pixelated graphic or a barcode, where you want hard edges.

Reusing one image many times

Every Image element embeds its own XObject, so adding the same photograph to fifty pages through fifty elements embeds it fifty times. When an image repeats — a logo in a running header, a background on every page — register the XObject once and draw it by name instead.

use Papier\Graphics\Image\PngImage;
use Papier\Content\ContentStream;

$logo = PngImage::fromFile(__DIR__ . '/logo.png');

$doc->header(function ($page, int $n, int $total) use ($logo) {
    $page->getResources()->addXObject('Logo', $logo->getStream());

    $cs = new ContentStream();
    $cs->save()->transform(90, 0, 0, 30, 72, 795)->drawXObject('Logo')->restore();
    $page->addContent($cs);
});

Because getStream() returns the same stream object each time, the writer emits it once and every page references it. See Documents & pages for running headers and footers.

Inline images

PDF also allows an image to be written directly into a content stream rather than as a separate XObject, using the BI … ID … EI operators. The saving is the object overhead, so it only pays for genuinely tiny images — a few dozen bytes of icon or a one-pixel spacer.

$cs->save()
   ->transform(20, 0, 0, 20, 72, 400)
   ->inlineImage([
       'W'   => 4,               // Width
       'H'   => 4,               // Height
       'CS'  => 'G',             // ColorSpace: DeviceGray
       'BPC' => 8,               // BitsPerComponent
   ], $rawSamples)
   ->restore();

Inline image dictionaries use abbreviated keys, as shown. The data is uncompressed unless you specify a filter, and the spec caps inline images at 4 KB of data — beyond that an XObject is required. For anything you would recognise as a picture, use the Image element.

CCITT Group 4 for scanned documents

Bilevel scans — faxes, signed contracts, archived paperwork — compress far better with CCITT Group 4 than with Flate, because the algorithm is built around runs of black and white on a page rather than general-purpose byte patterns. Ratios around 10× on typical scans are normal.

use Papier\Filter\CCITTFaxDecode;
use Papier\Objects\{PdfDictionary, PdfInteger};

$params = new PdfDictionary();
$params->set('K', new PdfInteger(-1));          // -1 = pure Group 4
$params->set('Columns', new PdfInteger($cols));
$params->set('Rows', new PdfInteger($rows));

$codec   = new CCITTFaxDecode();
$encoded = $codec->encode($raw1bpp, $params);
$decoded = $codec->decode($encoded, $params);   // lossless round trip

The input is 1 bit per pixel packed MSB first, one row at a time, padded to a byte boundary at the end of each row. Embedding the result means building the image XObject by hand, because the data is already compressed and must not be filtered again:

use Papier\Objects\{PdfName, PdfStream};

$img = new PdfStream();
$d   = $img->getDictionary();
$d->set('Type', new PdfName('XObject'));
$d->set('Subtype', new PdfName('Image'));
$d->set('Width', new PdfInteger($cols));
$d->set('Height', new PdfInteger($rows));
$d->set('BitsPerComponent', new PdfInteger(1));
$d->set('ColorSpace', new PdfName('DeviceGray'));
$d->set('Filter', new PdfName('CCITTFaxDecode'));
$d->set('DecodeParms', $params);
$img->setData($encoded);

$page->getResources()->addXObject('Fax', $img);

$cs = new ContentStream();
$cs->save()->transform(360, 0, 0, 240, 100, 480)->drawXObject('Fax')->restore();
$page->addContent($cs);

The decoder works on existing files too, so scanned PDFs from third parties can be read back. See examples/31_ccitt_fax.php, which prints the compression ratio and verifies the round trip.

Other image filters

Papier passes DCT (JPEG), JBIG2 and JPX (JPEG 2000) data through unchanged when reading and writing, so images in those formats survive a merge or an import without being recompressed. Only CCITT and the general-purpose filters — Flate, LZW, ASCIIHex, ASCII85, RunLength — are implemented as codecs. See File structure.

Extracting images from a PDF

Reading works in the other direction as well:

use Papier\Parser\PdfParser;

$parser = new PdfParser(file_get_contents('scanned.pdf'));
$parser->parse();

foreach ($parser->extractImages() as $image) {
    // width, height, colour space, bit depth, filter and raw data
}

See Reading PDFs for the full parser API, and examples/04_images.php for a page exercising JPEG, PNG, alpha, scaling, the fit helpers, opacity and fromFile() together.
