Documents & pages

Every Papier document starts with a PdfDocument and ends with a call to save(). In between you add pages, draw on them, and attach document-level features such as metadata, page labels, named destinations and attachments.

Creating a document

PdfDocument::create() builds an empty document with a catalog, an empty page tree and an Info dictionary. The optional argument is the PDF header version written to the file; the default of 1.7 matches ISO 32000-1 and is what you want unless you deliberately need to advertise an older or newer version.

use Papier\PdfDocument;

$doc = PdfDocument::create();          // header version 1.7
$doc2 = PdfDocument::create('1.4');    // advertise PDF 1.4

The version in the header is only a declaration. Papier still writes whatever structures you ask for, so if you enable object streams (a PDF 1.5 feature) while declaring 1.4, strict validators will complain. Leave the default alone in normal use.

Document metadata

The Info dictionary carries the human-readable description of the file that viewers show in their "Document properties" panel. Papier exposes it through five chainable setters. Each returns the document, so they compose into a single statement.

$doc->setTitle('Quarterly Report')
    ->setAuthor('Jane Doe')
    ->setSubject('Q3 revenue by product line')
    ->setKeywords('revenue, quarterly, 2026')
    ->setCreator('Acme Reporting Pipeline');

Setting a title is worth doing even for throwaway files: combined with the displayDocTitle() viewer preference described below, it is what appears in the reader's window title bar instead of the filename. If you need the raw dictionary — for instance to inspect what will be written — $doc->getInfo() returns the DocumentInfo object.

Papier also emits an XMP metadata packet alongside the Info dictionary. Archival profiles such as PDF/A require the two to agree, which Papier handles for you when you call enablePdfA(). See File structure.

Adding pages

addPage() appends a blank page and returns the PdfPage object so you can immediately draw on it. The two arguments are width and height in points, where one point is 1/72 inch. The default is ISO A4.

$page = $doc->addPage();                  // A4: 595.28 × 841.89 pt
$wide = $doc->addPage(1000, 500);         // custom size

Pages are appended in call order, which is also the order they appear in the finished file. The coordinate origin is the bottom-left corner and the y axis grows upward — the opposite of most screen-drawing APIs. A y of 750 on an A4 page is near the top.

Standard page sizes

PdfPage exposes the common paper sizes as two-element constants, which you spread into addPage() with the argument-unpacking operator.

use Papier\Structure\PdfPage;

$a4      = $doc->addPage(...PdfPage::SIZE_A4);
$letter  = $doc->addPage(...PdfPage::SIZE_LETTER);
$tabloid = $doc->addPage(...PdfPage::SIZE_TABLOID);

Orientation and rotation

There are two distinct ways to get a landscape page, and they are not interchangeable.

setLandscape() swaps the width and height of the page box. The page genuinely becomes wider than it is tall, and your drawing coordinates follow: x now runs to 841.89 on an A4 sheet.

$page = $doc->addPage()->setLandscape();   // 841.89 × 595.28

setRotation() leaves the page box alone and sets the /Rotate entry, which instructs the viewer to rotate the rendered page clockwise when displaying or printing it. Your content is authored in the unrotated coordinate system. The value must be a multiple of 90.

$page = $doc->addPage();
$page->setRotation(90)
     ->add(Text::write('Rendered sideways')->at(72, 300)->font($body, 14));

Use setLandscape() when you are laying out a landscape document from scratch, and setRotation() when you need to reorient existing content — for example a portrait page imported from another file that should print sideways.

Page boundary boxes

Beyond the media box set by the page dimensions, a page can declare four further boxes used by prepress and viewing tools. All four take the two opposite corners of a rectangle.

$page->setCropBox(0, 0, 595.28, 841.89)   // visible region
     ->setBleedBox(9, 9, 586, 833)        // production clipping
     ->setTrimBox(18, 18, 577, 824)       // finished page after trimming
     ->setArtBox(36, 36, 559, 806);       // meaningful content extent

If you are producing files for a commercial printer, the trim box is the one they will ask about — it defines where the guillotine cuts, and the bleed box tells them how far artwork must extend past that cut.

Drawing on a page

PdfPage::add() accepts any number of elements and renders them into the page's content stream. Elements are the high-level fluent API: Text, TextBox, Image, Rectangle, Circle, Line and Table.

use Papier\Elements\{Color, Rectangle, Text};

$font = $doc->addFont('Helvetica-Bold');

$page->add(
    Rectangle::create(0, 800, 595, 42)->fill(Color::rgb(0.12, 0.22, 0.42)),
    Text::write('Annual Report')->at(36, 815)->font($font, 16)->color(Color::white()),
);

Elements are drawn in the order you pass them, so later elements paint over earlier ones. See Text & fonts, Graphics, Images and Tables for the full element catalogue.

Multiple content streams

A PDF page may hold an array of content streams rather than a single one; viewers concatenate them before interpreting. Papier exposes this through addContent(), which appends a ContentStream you have built yourself with raw PDF operators.

use Papier\Content\ContentStream;

$cs = new ContentStream();
$cs->save()
   ->setFillRGB(0.9, 0.3, 0.2)
   ->drawCircle(300, 400, 60)
   ->fill()
   ->restore();

$page->addContent($cs);

Mixing add() and addContent() on the same page is normal and is exactly what the bundled examples do — elements for text and boxes, a hand-written stream for the shadings and patterns that have no element equivalent. The two interleave in call order.

Splitting content across streams is also a practical way to keep each stream small, since every stream is compressed independently. getContentStreams() returns the current array and getContentStreamCount() its length; truncateContentStreams($n) discards everything after the first $n streams, which is how the running-element machinery rolls back a speculative render.

Running headers and footers

A running header or footer has an awkward dependency: the text "Page 3 of 12" cannot be written until every page exists. Papier resolves this by deferring the render. You register a closure, and Papier calls it once per matching page immediately before writing the file, when the total page count is finally known.

use Papier\Structure\PdfPage;
use Papier\Elements\{Color, Line, Text};

$doc->header(function (PdfPage $p, int $n, int $total) use ($body) {
    $p->add(
        Text::write('Quarterly Report')->at(72, 800)->font($body, 9)->color(Color::gray(0.4)),
        Line::from(72, 794)->to(523, 794)->color(Color::gray(0.8))->width(0.5),
    );
});

$doc->footer(function (PdfPage $p, int $n, int $total) use ($body) {
    $p->add(Text::write("Page $n of $total")->at(480, 30)->font($body, 9));
});

The closure receives the page object, the 1-based page number, and the total page count. header(), footer() and the generic onEachPage() are all the same mechanism with different names; the distinction is documentary, letting you read back which callbacks were intended as chrome. You may register as many of each as you like and they run in registration order.

Selecting pages with PageRule

The second argument of header(), footer() and onEachPage() restricts which pages the callback applies to. It accepts a PageRule enum case, an integer, or a closure.

use Papier\PageRule;

$doc->footer($render);                          // PageRule::All (default)
$doc->onEachPage($notice,     PageRule::First); // first page only
$doc->onEachPage($colophon,   PageRule::Last);  // last page only
$doc->onEachPage($confidential, PageRule::Odd); // 1, 3, 5, …
$doc->onEachPage($tab,        PageRule::Even);  // 2, 4, 6, …
RuleApplies to
PageRule::AllEvery page (the default)
PageRule::OddPage numbers 1, 3, 5, …
PageRule::EvenPage numbers 2, 4, 6, …
PageRule::FirstThe first page only
PageRule::LastThe last page only

Odd and Even are the natural choice for double-sided printing, where the outer margin alternates sides. When a fixed rule is not enough, pass an integer for every Nth page, or a closure receiving the page number and total that returns a boolean.

$doc->onEachPage($sectionRule, 5);   // pages 5, 10, 15, …

$doc->onEachPage(
    $watermark,
    fn(int $n, int $total): bool => $n > 1 && $n < $total,   // skip cover and colophon
);

See examples/33_page_operations.php for a five-page report with a running header, a "Page X of Y" footer, and a confidential notice restricted to odd pages.

Page operations

Three static helpers on PdfDocument take existing files and produce a new one. They parse the sources, wrap each page as a form XObject, and place it on a fresh page — so the visual result is preserved but interactive features on the source pages (annotations, form fields) are not carried across.

Merging documents

PdfDocument::merge(['cover.pdf', 'body.pdf', 'appendix.pdf'], 'combined.pdf');

Pages are concatenated in the order the source paths are given.

Extracting and reordering pages

PdfDocument::extractPages('report.pdf', [2, 4], 'excerpt.pdf');
PdfDocument::extractPages('report.pdf', [3, 1, 2], 'reordered.pdf');

Page numbers are 1-based and are honoured in the order supplied, so the same call both subsets and reorders. Out-of-range numbers are skipped rather than raising an error.

N-up imposition

nUp() tiles several source pages onto each output sheet — the usual way to produce handouts or proof sheets. Each source page is scaled to fit its cell while preserving aspect ratio, then centred within it.

PdfDocument::nUp('report.pdf', 2, 2, 'handout.pdf');   // 4 pages per sheet

PdfDocument::nUp(
    'report.pdf',
    cols:   3,
    rows:   3,
    outPath:'nine-up.pdf',
    sheetW: 841.89,   // A4 landscape
    sheetH: 595.28,
    margin: 24.0,
    gutter: 12.0,
);

Cells are filled left to right, top to bottom. The sheet defaults to A4 portrait with a 20 pt outer margin and a 10 pt gutter between cells.

Importing pages into a document

The static helpers above are convenience wrappers around importPages(), which brings pages from another PDF into a document you are still building. Because it returns the newly created PdfPage objects, you can then draw over the imported content — the basis for stamping, watermarking and filling pre-printed forms.

$doc = PdfDocument::create();
$font = $doc->addFont('Helvetica-Bold');

$pages = $doc->importPages('contract.pdf');           // all pages
$some  = $doc->importPages('contract.pdf', [1, 3]);   // selected pages

$pages[0]->add(
    Text::write('DRAFT')->at(200, 400)->font($font, 60)->opacity(0.15),
);

$doc->save('stamped.pdf');

For finer control, importPage() takes a single prepared ImportedPage plus an offset and uniform scale factor, which is what you need to place a source page at a specific position on a larger sheet. See examples/16_import_pages.php.

Page labels

Page labels change the numbering that the viewer shows in its thumbnail strip and "go to page" box, without affecting the physical page order. The classic use is front matter numbered in lowercase roman numerals followed by a body restarting at 1.

use Papier\Structure\PageLabelStyle;

$doc->addPageLabel(0, PageLabelStyle::RomanLower);   // i, ii, iii, iv
$doc->addPageLabel(4, PageLabelStyle::Decimal);      // 1, 2, 3, …

The first argument is a 0-based page index at which the range begins; the range runs until the next one starts. A range may also set its own starting number and a prefix.

$doc->addPageLabel(4, PageLabelStyle::Decimal, startValue: 1, prefix: 'p.');
$doc->addPageLabel(20, PageLabelStyle::AlphaUpper, startValue: 1, prefix: 'Appendix ');

Named destinations

A named destination is a symbolic label for a position in the document. Link annotations, bookmarks and external URLs can target it by name, so the target survives content being reflowed onto different pages.

$intro = $doc->addPage();
$doc->addNamedDestination('intro', $intro, 72, 750);
$doc->addNamedDestination('chapter-2', $chapter2Page, 0, 780, 1.0);

The three optional arguments are the left coordinate, the top coordinate and the zoom factor. Passing null for any of them means "leave that aspect of the current view unchanged", which is usually what you want for the zoom — forcing a zoom level overrides the reader's own preference. A zoom of 0 also means unchanged.

See Navigation for linking to these destinations from bookmarks and link annotations.

Open action

The open action tells the viewer where to position the document when it is first opened. Without one, readers start at the top of page 1.

$doc->setOpenDestination($page, 0, 841);         // top of a given page
$doc->setOpenDestination($summaryPage, 72, 500); // scrolled to a specific spot

Use it to open a long report at its executive summary, or to skip a cover sheet. As with named destinations, leaving the zoom as null respects the reader's default magnification.

Viewer preferences

Viewer preferences are hints about how the document should be presented — which chrome to hide, how to lay out the window, and how to print. Readers are free to ignore them, but the mainstream ones honour the common keys.

use Papier\Viewer\{ViewerPreferences, PrintScaling, Duplex, NonFullScreenPageMode, ReadingDirection};

$doc->setViewerPreferences(
    ViewerPreferences::create()
        ->displayDocTitle()
        ->hideToolbar()
        ->fitWindow()
        ->centerWindow()
        ->printScaling(PrintScaling::None)
        ->duplex(Duplex::FlipLongEdge)
        ->nonFullScreenPageMode(NonFullScreenPageMode::UseOutlines)
        ->readingDirection(ReadingDirection::LeftToRight)
        ->numCopies(2)
);

The boolean flags — hideToolbar(), hideMenubar(), hideWindowUI(), fitWindow(), centerWindow(), displayDocTitle() — default to true, so calling them with no argument turns them on. Pass false to write an explicit negative.

Two are worth singling out. displayDocTitle() makes the reader show your setTitle() value in the window title instead of the filename. printScaling(PrintScaling::None) disables the reader's "fit to printable area" shrink, which matters whenever the document must print at exact physical size — floor plans, forms, anything with a scale bar.

EnumCases
PrintScalingNone, AppDefault
DuplexSimplex, FlipShortEdge, FlipLongEdge
NonFullScreenPageModeUseNone, UseOutlines, UseThumbs, UseOC
ReadingDirectionLeftToRight, RightToLeft

Less common keys such as /ViewArea or /PrintPageRange have no dedicated method; write them directly with set($key, $value), passing a PDF object.

File attachments

A PDF can carry arbitrary embedded files, which readers surface in an attachments or paperclip panel. This is the standard way to ship the machine-readable source of a report alongside its human-readable rendering.

$doc->attachFile('data.csv', file_get_contents('report.csv'), 'text/csv');
$doc->attachFile('invoice.xml', $xmlString, 'application/xml');
$doc->attachFile('readme.txt', 'Figures are in thousands of USD.', 'text/plain');

The first argument is the display name shown to the user, the second the raw bytes, and the third the MIME type — it defaults to application/octet-stream, which makes readers treat the file as an opaque download. Supplying a real type lets them offer a sensible "open with".

Attachments are stored uncompressed relative to their own format, so embedding large binaries inflates the PDF by roughly their own size. Compress before attaching if size matters.

Attachments can be read back out of an existing file with PdfParser::getAttachments(); see Reading PDFs.

Saving the document

save() serialises the document — running any deferred header and footer callbacks first — and writes it to disk.

$doc->save(__DIR__ . '/output/report.pdf');

Two alternatives exist for when a file on disk is not what you want. toString() returns the complete PDF as a byte string, which is what you need to hand the bytes to a parser, a signer, or a cloud storage client. output() sends the file to the browser with the appropriate headers.

$bytes = $doc->toString();

$doc->output('report.pdf');          // inline, rendered in the browser
$doc->output('report.pdf', false);   // as a download attachment

Producing smaller files

Calling useObjectStreams() before saving packs non-stream objects into compressed object streams and writes a cross-reference stream instead of a classic table. On documents with many small objects this typically cuts file size noticeably, at the cost of requiring a PDF 1.5 reader.

$doc->useObjectStreams();
$doc->save('small.pdf');

See examples/17_compressed_objects.php for a size comparison, and File structure for what the two layouts actually look like on disk.
